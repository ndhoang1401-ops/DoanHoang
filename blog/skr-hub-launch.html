Building skr-hub: From Idea to Launch in 30 Days

Jan 15, 2024 • 8 min read • Full-Stack Development

30 days. That's all it took to go from a blank repository to a fully functional, deployed full-stack portfolio platform. This is the story of skr-hub — how we built it, what we learned, and the challenges we faced.

The Motivation

I needed a portfolio that could showcase my work while demonstrating my technical skills in real-time. Not just a static website, but a living, breathing application that could handle:

  • Complex animations and visual effects (Three.js particles)
  • Real-time data fetching (GitHub API integration)
  • Form handling and email notifications
  • Multiple theme support
  • Lighthouse performance targets (95+)

The Tech Stack

We chose a modern, proven tech stack:

  • Frontend: React with GSAP for animations
  • Backend: Node.js with Express for API routes
  • Database: MongoDB for flexible data storage
  • 3D Graphics: Three.js for particle effects
  • Deployment: Render for serverless hosting
  • Email: EmailJS for contact form submissions

Week 1: Planning & Setup

The first week was all about planning. We created wireframes, defined the project structure, and set up the development environment. Key decisions made:

  • Monorepo structure for client and server
  • Environmental variables for API keys and secrets
  • Git workflow with feature branches
  • CI/CD pipeline with GitHub Actions

Week 2: Frontend Development

With the foundation in place, we moved to building the UI components:

  • Created reusable React components (Hero, ProjectCard, SkillBar)
  • Implemented GSAP animations for smooth transitions
  • Built the Three.js particle field from scratch
  • Designed the glassmorphism UI with CSS modules

One major challenge: Three.js performance. We had to optimize particle count based on device capabilities, reducing from 2000 particles on desktop to 600 on mobile.

Week 3: Backend & Integration

The backend was critical for real-time features:

  • Built REST API endpoints for project data and stats
  • Integrated GitHub API to fetch repository stars and commits
  • Set up EmailJS for contact form notifications
  • Added error handling and request validation

Performance was a concern from day one. We implemented caching strategies and lazy-loading to keep API response times under 100ms.

Week 4: Optimization & Deployment

The final week was all about optimization and getting live:

  • Lighthouse audit: achieved 95+ scores across all metrics
  • Image optimization with WebP format
  • Code splitting to reduce bundle size
  • Service worker implementation for offline support
  • Deployed to Render with automatic CI/CD

Key Learnings

Performance is a feature. Users notice fast websites. We obsessed over load times and achieved sub-2s first paint through aggressive optimization.

Mobile-first matters. Building for mobile constraints (smaller GPU, limited bandwidth) forced better decisions for desktop as well.

Animations aren't free. Every animation must justify its existence. We cut animations that didn't add value, even if they looked cool.

APIs are real bottlenecks. GitHub API rate limits and latency were more challenging than expected. Caching became critical.

Deployment Strategy

We chose Render for its simplicity and automatic deployments from Git. The workflow:

  1. Push code to main branch
  2. GitHub Actions run tests
  3. Render automatically builds and deploys
  4. Service worker updates on next visit

Metrics

After 30 days, here's what we achieved:

  • Lighthouse Score: 95/100 🚀
  • First Contentful Paint: 1.2s
  • Time to Interactive: 2.8s
  • Bundle Size: 145KB (gzipped)
  • Mobile Performance: 90+

Conclusion

Building skr-hub in 30 days taught me that constraints breed innovation. By focusing on what matters (performance, UX, code quality), we shipped a product we're genuinely proud of.

Next steps? We're planning to add a blog CMS, project filtering, and a contact form backend. But the foundation is solid, and the journey doesn't end here.
