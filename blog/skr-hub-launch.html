Building skr-hub:
From Idea to Launch in 30 Days

Jan 15, 2024 · 8 min read · Nguyễn Doãn Hoàng
React Node.js Three.js Performance Deployment

30 days. That's all it took to go from a blank repository to a fully functional, deployed full-stack portfolio platform. This is the story of skr-hub — how it got built, what broke, what we learned, and why Lighthouse 95+ is achievable if you actually care.

The Motivation

I needed a portfolio that could demonstrate technical skill, not just describe it. Not a static website with pretty colours — a living application that actually works under pressure. That meant:

  • Complex animations and 3D visual effects in real-time (Three.js particles)
  • Real-time data from external APIs (GitHub stars, commit counts)
  • Contact form with email delivery, not just a mailto link
  • Multiple themes with persistent user preference
  • Lighthouse performance score of 95+ — non-negotiable

The rule I set myself: if I'm going to call myself an engineer, the portfolio itself has to be evidence. Not just projects I link to — the site itself.

The Tech Stack

Chosen for real reasons, not hype:

React Node.js + Express MongoDB Three.js Render EmailJS

React for component reuse. Node for API flexibility. Three.js because static doesn't cut it. Render because it deploys from Git with zero config. EmailJS because building a mail server for a portfolio is overkill.

Week by Week

Week 1

Planning & Foundation

Wireframes, project structure, environment setup. Monorepo for client + server. Git workflow with feature branches. CI/CD pipeline via GitHub Actions from day one — not bolted on later.

Week 2

Frontend — UI & Animation

Built all React components: Hero, ProjectCard, SkillBar. Implemented GSAP for scroll-triggered transitions. Built the Three.js particle field from scratch — initial count of 2,000 particles on desktop, dropped to 600 on mobile after GPU profiling.

Week 3

Backend & Integration

REST API endpoints for project data and live stats. GitHub API integration with a caching layer (critical — rate limits hit hard in testing). EmailJS wired up for contact form. Error handling and input validation throughout.

Week 4

Optimisation & Deploy

Lighthouse audits every day. WebP image conversion. Code splitting to cut bundle size. Service worker for offline support. Final deployment to Render with automatic builds on push to main.

The Challenges That Actually Hurt

Three.js on Low-End Devices

The particle system looked great on my MacBook. On a mid-range Android phone? Slideshow. Solution: detect GPU tier via gl.getParameter(), scale particle count dynamically (2000 → 800 → 400), and use requestAnimationFrame with frame-skipping on slow devices.

GitHub API Rate Limits

Unauthenticated: 60 requests/hour. With a portfolio getting 200+ visits/day, that's a wall. Solution: server-side caching with a 15-minute TTL. Requests hit the cache, not GitHub. API calls dropped from hundreds to single digits per hour.

Lighthouse on a Budget

Getting to 95+ meant cutting everything that couldn't justify its weight:

  • Removed 3 fonts, kept 2 with font-display: swap
  • Replaced all top/left animations with transform — GPU compositing only
  • Lazy-loaded all images below the fold
  • Preloaded critical CSS and fonts
  • Eliminated every layout shift with explicit width/height on images

Key Learnings

Performance is a feature, not a phase. Building for performance from day one is 10× easier than retrofitting it in week 4.

Mobile-first forced better decisions everywhere. Every GPU constraint, bandwidth limit, and touch-target size I fixed for mobile made the desktop experience cleaner too.

Animations are not free. I cut animations that looked cool but added no clarity. If it doesn't help the user understand something faster, it's decorative weight.

Caching is not optional. Any API you don't control will betray you at peak load. Cache aggressively, invalidate carefully.
